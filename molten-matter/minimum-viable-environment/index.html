Minimum viable environment

2019/05/13

I have some basic familiarity with most of the mainstream development environments around these days. I started out with Vim, but I managed to live with Emacs for a while, and nowadays I mostly use VS Code and I know the basics of Visual Studio and some JetBrains IDEs. To complement these I use bash and Powershell, despite not being a huge fan of either.

Most of these tools have wildly different interaction modes and differ a lot in their finer details, so to stay sane I always install a Vim emulation layer when setting them up on a new machine: going by popularity it’s Evil for Emacs, VsCodeVim for VS Code, VsVim for Visual Studio, and whatever the IntelliJ one is called. (By the way, I really appreciate that IntelliJ asks you whether you want to install its Vim plugin in the first-time setup wizard.)

Of course you can’t perfectly emulate Vim on anything that isn’t Vim itself, and even on the infinitely extensible Emacs you’ll eventually run into some issues with plugins that don’t play well with it. Honestly I’m ok with this; Vim runs on all the platforms I could possibly care about and if I choose to use something different it’s because I find value in the features of that environment that wouldn’t work as well on Vim.

So the bone to pick I have with these emulation plugins is not that they don’t emulate Vim to perfection, but that they try too hard to.

Faking vim

VsCodeVim actually keeps its own undo history. The implementation is really complex: it has to comply to whatever it is that vim does to the keystroke, or else angry mobs will show up in the issue tracker to complain. Because of this complexity and of Vim’s compositional nature any bug in the implementation will likely break a bunch of other features, and sure enough there’s no shortage of bugs related to the undo stack in the issue tracker.

I prefer to just turn it off.

Another thing that bother me is that the ex commands show up in VS Code’s tiny status bar rather than in the command palette where they would fit much better. Apparently it used to work this way but was changed due to limitations in the extension API. Search is also awkwardly implemented as a completely separate function from VS Code’s native search, so I can use it to quickly jump to something but it’s useless to visually scan the file because it doesn’t hightlight matches.
